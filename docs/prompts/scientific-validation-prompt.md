# LLM Prompt — Scientific-Consistency Validation Scheme for FatigueMeter

Paste the prompt below into a capable coding LLM with this repository as context. Its job is to produce an **automated consistency harness** that continuously checks the FatigueMeter app's computed values for **internal consistency with the project's stated model** in `docs/literature-review.md`, `docs/white-paper.md`, and `docs/references.md`.

> **Epistemic status — read this first (Rev 2).** This harness verifies **"the app does not contradict the project's stated model,"** which is valuable **regression protection**. It is **NOT** a proof that "the app cannot contradict the science": the documents it checks against are authored by this same project and contain synthesis and speculation, so if a synthesized assumption is wrong, the harness will happily certify an app that is wrong in exactly that way. **Self-consistency ≠ external validity.** Only real-file testing against *measured* fatigue criteria (the criterion-validity pilot in white-paper §10) can establish external validity, and even that is limited by the absence of an on-bike fatigue ground truth. State this caveat in the harness's own report header.

This is deliberately separate from ordinary unit tests. Unit tests check that a formula is coded correctly; this harness checks that the *system's behavior and outputs stay internally consistent with the documented model* across realistic and adversarial ride scenarios.

---

## PROMPT

You are a verification engineer with a strong background in exercise physiology and signal processing. Build a **model-consistency validation harness** for the FatigueMeter application (a cyclist durability/fatigue *estimator*; see `docs/white-paper.md` for the model and `docs/literature-review.md` + `docs/references.md` for the evidence base). The harness asserts that FatigueMeter's outputs remain **internally consistent with the documented model**, and must clearly separate **hard invariants** (violation = bug) from **soft/plausibility checks** (violation = warning to review) from **calibration-dependent** behavior. Every check must be labeled with its tier and must state, in the report, that plausibility checks verify *implementation-against-spec*, not *agreement-with-measurement*.

### Deliverable
A runnable test suite (choose Python for off-device testing of the extracted pure functions; the app's formulas should be provided as pure functions per the generation prompt, or reimplemented faithfully and cross-checked). Include: synthetic signal generators, real-file ingestion (FIT/CSV with power, HR, RR, cadence), a property-based testing layer, an assertion catalog, and a human-readable report that cites which consensus statement each check enforces (link back to `docs/references.md` entries).

### 1. Hard invariants (must always hold; violation = fail)
Encode these as assertions over any run:
- **Definitional identities:** `TSB == CTL − ATL` exactly (using yesterday's values); `IF == NP/FTP`; `TSS == duration_h · IF² · 100` within float tolerance; NP = 4th-root of 30 s rolling mean of power⁴. **(Report note:** the NP identity is checked for *coding correctness only* — the white paper flags rolling NP over short windows as unvalidated-at-granularity, so a passing check does not endorse the construct.)
- **Bounds:** AFI ∈ [0,100]; decoupling % real-valued; EWMA states non-negative for non-negative TSS. **DFA-α1: finite and > ~0.2 whenever the gate passes; the UPPER bound is a SOFT/WIDE check only** (resting/very-easy α1 legitimately approaches/exceeds ~1.5–1.6 with strongly-correlated dynamics, so a hard upper bound would false-fail correct ride-start physiology).
- **Filter type:** the estimator is a **linear (time-varying) Kalman filter** — the model is linear in the state (all nonlinearities are functions of the measured input `P`). Do not assume/require an EKF; if the code ships an EKF, assert its Jacobian reduces to the constant transition matrix (else the model was mis-specified).
- **Ordering of concerning bands:** the AFI/decoupling/α1 cutoffs are strictly ordered (fresh < accumulating < high < severe); no overlapping or inverted bands.
- **Artifact gate:** DFA-α1 is emitted only when window artifact ≤ the configured threshold (default 5%); otherwise the value is withheld and the fallback path is active.
- **Kalman sanity:** covariance stays positive-definite; state estimates do not diverge (no NaN/Inf) under sensor dropouts, RR gaps, power=0, HR=0.
- **Ledger integrity:** CTL/ATL update is idempotent per day and survives a simulated mid-ride crash without corruption.

### 2. Consensus-plausibility checks (monotonicity & direction; violation = warning)
Drive the model with synthetic and real rides and assert the *directions* the literature establishes:
- **α1 vs intensity (ENSEMBLE-MEAN ONLY, wide tolerance).** Across a *population/ensemble* of synthetic rides, mean DFA-α1 **decreases** as intensity rises, crossing ~0.75 near AeT and ~0.5 near AnT (Rogers/Gronwald). **Do NOT assert per-run monotonicity:** the only cycling study found failure across a *range* of α1 values and non-monotonic individual behavior once fatigued, so a per-run monotonicity assertion could flag *correct* physiology as a violation. Use wide tolerance and mean-only; this is a soft check.
- **Decoupling under drift:** for a long steady sub-threshold effort with imposed cardiac drift, decoupling % is **non-negative and generally increases** over time (after the steadiness gate); for a fresh short effort it stays <5%. Tolerate the low-SNR reality (≈2% mean, high SD at 75% FTP) — do not assert a tight trajectory.
- **α1 within-ride fatigue drift (ensemble-mean, soft):** across an ensemble of prolonged fixed-power efforts, mean DFA-α1 tends **downward**; do not require it of every individual run.
- **Durability magnitude:** the modeled VT1/decoupling drift after ~1,400–1,680 kJ is in the ballpark of **~6–10%** (Maunder/Stevens); flag if the model predicts implausibly small (<1%) or large (>25%) drift for a typical trained profile.
- **Sub-CP drift is ALLOWED (Rev 2 — replaces the deleted CP-gate invariant).** The earlier "assert `F` does not grow sub-CP" check was **wrong and is removed**: cardiovascular drift occurs during prolonged sub-CP/Z2 riding (thermoregulation, plasma-volume loss), and the supporting data were collected at 75% FTP (below CP). Instead assert only that `F`'s charge is **graded** (larger above AeT, non-zero but small below) and monotonically related to intensity+duration — never that it is zero sub-CP. A harness that rejects real Z2 drift is inverting its purpose.
- **α1↔F coupling is wired (Rev 2, structural).** Assert that injecting an α1 measurement *below* its power-predicted `A1_target` **moves the `F` estimate** (and thus AFI) — i.e., DFA-α1 genuinely informs the fatigue state. If α1 innovations leave `F` unchanged, the "fusion" claim is violated (this was a real defect in Rev 1).
- **`F` observability = MATHEMATICAL CONDITIONING, not physiological identifiability (Rev 3):** run a simulated test — generate a ground-truth `F(t)` with realistic variable-power + noise, run the filter, assert `F` is recovered within tolerance on **variable-power** profiles, and assert/report that on **constant-power** profiles `F` is only weakly recovered (prior-dominated). **Report must state:** this proves recoverability *under the assumed model only* (non-degenerate observability Gramian), NOT that real drift decomposes into fatigue vs thermal/dehydration the way the model assumes — that needs the external pilot.
- **Respiration/artifact does NOT manufacture fatigue (Rev 3, required):** inject a **respiration-only α1 excursion at constant power and constant true fatigue** (and separately an RR-artifact burst); assert `F`/AFI does **not** materially rise — i.e. `R_A1` inflation on rapid-fB/high-artifact windows actually de-weights those excursions. Without this the α1↔F coupling can fabricate fatigue from breathing.
- **Long steady Z2 → moderate, not severe (Rev 3):** a 3–4 h steady sub-threshold ride must yield AFI in a **moderate** band, not saturated/severe — verifies the graded charge (`κ_i` onset at P_AeT + standing `κ_d`) is tuned consistently with the end-of-*hard*-ride `F_ref`.
- **Recovery relaxes `F` (Rev 3):** on a recovery/coasting/stopped segment, assert AFI does **not** increase (`κ_d` gated on active; `F` decays via `τ_rec`).
- **Feat vs Attrition = evidence, NOT a gate (Rev 2, soft):** on a synthetic **hard-effort ride** assert FeatScore dominates; on a synthetic **long grind** assert AttritionScore dominates; assert a W′ "match" is counted only on a genuine W′bal <threshold→recovery cycle. **Do NOT assert that the classifier gates the status band** — per §8.2 it is off the critical path; assert instead that the status remains **descriptive** and the classification is shown as context only.
- **No imperative language (Rev 3 — check MOOD, not a string blacklist):** assert no status/advisory string is in the **imperative mood** (no directive verb — not just "TURN BACK"/"STOP" but also "EASE SOON", "back off", etc.). Maintain an allowed-copy list of descriptive phrasings and assert the UI strings are drawn from it; a substring blacklist is insufficient. The advisory carries a persistent "advisory · not validated" tag on the band itself.
- **Numeric-AFI release gate (Rev 3):** assert that **pre-pilot** (no positive criterion-validity result recorded) the UI exposes only the **3-state categorical** (green/amber/red) and coarse start/now — **no precise 0–100 AFI digit, no point-value start/now/end, no projected-end tick**; these unlock only when a positive-pilot flag is set (or when an explicit "ship-number-pre-pilot" override is recorded).
- **Training-load realism:** 1 h at FTP ⇒ ~100 TSS; a hard 3 h ride ⇒ 200–300 TSS; CTL ramp warnings fire when weekly CTL rise >5–8 points; ACWR >1.5 raises a *descriptive* (not predictive) warning.
- **TSB tapering behavior:** after a load reduction, ATL falls faster than CTL, so TSB rises (Banister τ2<τ1) — assert this on a simulated taper.

### 3. Adversarial / robustness scenarios
Generate and run these, asserting no hard-invariant violations and sensible fallback:
- Corrupt/ectopic RR streams at 3%, 6%, 12% artifact → α1 withheld above gate; fallback to decoupling-only engages and is surfaced in output.
- Wrist-optical-quality RR (jittered) → the harness confirms the app flags low confidence rather than emitting a confident α1.
- Power spikes/dropouts, HR sensor loss, pauses/stops, very short rides (<15 min, no baseline window) → no divide-by-zero, no NaN, graceful "insufficient data."
- Heat scenario (imposed extra drift) → decoupling AND α1 both move from the *shared* confound; assert the app **names the confound** and applies thermal suppression to **both** channels, and that the correlated agreement does NOT read as independent corroboration (the advisory should acknowledge shared-confound uncertainty, not fire confidently).
- Extreme profiles (very high/low FTP, HR_max) → outputs stay bounded and ordered.
- **Stale CP/W′ propagation:** feed a deliberately wrong CP → assert the downstream W′bal/match/FeatScore carry a data-quality caveat rather than silently driving the characterization.
- **Graceful-degradation matrix (Rev 3 — HARD requirement; exercise every row of white-paper §8.4).** For each dropped/absent/stale input — **power, HR, RR/HRV, cadence, stale FTP/CP, intermittent dropout, and total sensor loss** — assert: (a) **no exception escapes to the compute loop, no NaN/Inf, no blank screen/crash**; (b) every metric that does **not** depend on the missing input **keeps updating** (e.g. no power → HR/α1/RMSSD/HR-TRIMP still live; no HR → power/kJ/NP/W′bal/power-TSS still live; no RR → filter drops the α1 observation and blends decoupling-dominant, everything else unaffected); (c) affected tiles show an explicit "— / no <sensor>" availability marker, not a silent zero; (d) intermittent dropout **holds last-valid with a staleness timer** then marks unavailable, and reacquires cleanly; (e) under predict-only gaps (missing observation) the filter's covariance/uncertainty **widens** rather than freezing a confident stale value. The single glance screen must be **fully functional with any subset of sensors present.**

### 4. Provenance & honesty enforcement (unique to this project)
- **Convention/synthesis guard:** for every threshold the white-paper §9 provenance table marks as "convention" or "synthesis," assert the app treats it as a *configurable setting with the documented default*, and that changing the setting changes behavior (i.e. it is not hard-coded). Fail if a convention value is hard-coded as if validated.
- **Label enforcement:** assert that when calibration has not run, AFI / durability advisory / start-end fatigue are tagged "uncalibrated — estimate only"; that the fused AFI/advisory carries a **persistent "advisory · not a validated measurement" tag on the status band itself** (not only a footer); and that no output is labeled clinically meaningful. Assert the non-medical-device disclaimer is present.
- **Descriptive-not-imperative:** assert the durability advisory uses descriptive language only and never a directive verb; assert the Feat/Attrition classifier does not gate the status band.
- **No-overclaim check:** assert the app never emits the word "damage" as a certainty for the durability advisory, and that speculative items (glycogen flip, absolute-α1 alarm) are labeled speculative. Assert the report header states self-consistency ≠ external validity.
- **Traceability:** cross-check `docs/traceability.md` — every physiological constant used in code maps to a white-paper table row and a reference; fail if a constant has no provenance. **Report caveat:** the matrix is a stub — a passing check on an incomplete stub is not meaningful coverage; report the fraction of code constants actually present as rows, and treat missing `c_F` / `PROJECTED_AFI` / seeding-map rows as gaps to fill.

### 5. Calibration-dependence checks
- Assert that with **no** athlete calibration, the model uses literature defaults and says so; with a **calibration ride** applied, the personal power→DFA-α1 sigmoid is used only if its fit R² > 0.75, else it **falls back to decoupling-only and marks α1 display-only** (do not silently use a population sigmoid known to misfit many riders).
- Regression check (**threshold powers only**): after calibration, the α1-crossing powers and the durability kJ anchor move toward the athlete's **measured** AeT/AnT/durability. **Do NOT assert this for `F`/AFI:** there is no measured fatigue truth on-device, so calibration tunes threshold *crossings* and self-consistency, not the latent fatigue magnitude (white-paper §10). Any check implying AFI moves "toward measured fatigue" is invalid and must not be written.
- **Criterion-validity hook (documented, not asserted):** include a clearly-marked, skipped/`xfail` test stub for the external criterion-validity study (relate AFI/`F` to sustained-power decrement / lactate / RPE). **Use the correct statistics (Rev 3):** a **calibration/association analysis** (rank correlation + cross-validated calibration curve) — **not Bland–Altman**, which is invalid across mismatched units/constructs (AFI is a 0–100 index; the criteria are watts/mmol/RPE). Mark any n≈5 run as **proof-of-concept/feasibility, not validation**; a powered study is the release gate for the numeric AFI. The stub stays skipped until real data exist, so the owed validation is visible every run.

### 6. Reporting
Produce a report that, per check: states the assertion, the consensus statement it enforces (quoted/linked from `docs/references.md`), pass/fail/warn, and — for warnings — the observed vs expected direction/magnitude. Summarize hard-invariant failures at the top. The report must be readable by a coach or sports scientist, not just an engineer.

### Ground rules
- Treat `docs/literature-review.md`/`references.md` as the definition of *the project's stated model* (not as physiological ground truth); do not introduce thresholds not present there. Where the literature is explicitly uncertain (durability magnitude ranges, individual α1 variability ±0.28, running-vs-cycling α1 collapse, contested ACWR, low-SNR sub-threshold decoupling), encode **tolerant, ensemble-level** checks (ranges/directions), not brittle equalities or per-run monotonicity.
- Distinguish clearly, in both code and report, the three tiers: **hard invariant**, **plausibility (direction/range)**, **calibration-dependent**. A soft-check warning must never fail the build; a hard-invariant violation must.
- Where a check cannot be grounded in a cited consensus statement, mark it **heuristic** and keep it non-blocking.

Begin by enumerating the assertion catalog grouped by the six sections above, mapping each assertion to its `references.md` source and its tier, then implement the harness. Flag any assertion you cannot ground in the recorded literature.
